# Introduction

Welcome to the Orion documentation.

Orion is a modular business management platform designed to support service-oriented businesses and business groups. The platform supports the complete lifecycle of service delivery, from client acquisition through service execution to financial reporting and data integration within a single, coherent architecture.

Unlike traditional monolithic ERP systems, Orion is designed as a collection of independent business modules built on a common platform. This approach allows the system to evolve gradually while maintaining a consistent business model and user experience.

The primary objective of Orion is not merely to record business data, but to automate business processes, integrate information from multiple sources and provide reliable decision support.

# Documentation Structure

The documentation is organized into the following sections.

## Architecture

Defines the long-term architectural vision, engineering principles and domain model of Orion.

These documents describe **why** Orion is designed the way it is.

---

## Modules

Describes the business purpose, scope and responsibilities of each functional module.

Each module documents its business entities, business rules, dependencies and future development plans.

---

## Development

Contains practical information required for developing Orion.

Examples include:

- development environment
- coding standards
- Git workflow
- testing
- deployment

---

## Glossary

Provides authoritative definitions of business and technical terminology used throughout Orion.

All documentation should use these definitions consistently.

---

## Knowledge

Captures business knowledge that explains the reasoning behind the software.

Unlike technical documentation, these documents focus on accounting, consulting, reporting and business processes.

---

## Roadmap

Describes the planned evolution of Orion from the current foundation towards future releases.

---

# Documentation Principles

1. The documentation is maintained together with the source code.

2. Every significant architectural decision should be documented.

3. Business terminology should remain consistent across all documents.

4. Architecture documents describe concepts rather than implementation details.

5. Implementation details belong to developer documentation.

6. The documentation should evolve together with the software and always reflect the current state of the project.

---

# Contributing

When introducing new functionality, contributors are expected to update the relevant documentation as part of the same change.

No feature is considered complete until its implementation, tests and documentation are consistent.
